## 原則

コードレビューは脆弱性の最後の防衛線。本番到達前にリスクを検知し、修正を促す。
指摘時は「何がダメか」だけでなく「なぜダメか」「どう直すか」を併記する。

## チェックリスト

### 認証・認可
- [ ] **認証バイパスの有無**
  - 認証なしでアクセス可能なエンドポイントが意図的かを確認
  - なぜ: 認証漏れは最も深刻な脆弱性の一つ。不正アクセスに直結する
- [ ] **権限チェックの実装**
  - リソースアクセス時にユーザーの権限を検証しているか
  - なぜ: 認証済みでも他ユーザーのデータにアクセスできる（IDOR）リスクがある
- [ ] **CSRF対策の実装**
  - 状態変更を伴うリクエスト（POST/PUT/DELETE）にCSRFトークンまたはSameSite Cookie属性が設定されているか
  - なぜ: 攻撃者が被害者のセッションを悪用して意図しない操作を実行できる

### 入力バリデーション
- [ ] **ユーザー入力のサニタイズ**
  - 外部からの入力がDBクエリ・HTML出力・コマンド実行に渡る前にサニタイズされているか
  - なぜ: SQLインジェクション、XSS、コマンドインジェクションの原因となる
- [ ] **パラメータの型・範囲チェック**
  - 数値の範囲、文字列の長さ、許可された値のバリデーション
  - なぜ: 予期しない入力によるDoSやバッファオーバーフローを防ぐ

### 依存ライブラリ
- [ ] **既知脆弱性のチェック**
  - 新規追加・バージョン変更されたライブラリに既知の脆弱性がないか
  - コマンド例: `npm audit`（Node.js）、`pip audit`（Python）、`bundle audit`（Ruby）
  - なぜ: サプライチェーン攻撃の入り口になる
- [ ] **不要な依存の確認**
  - 使用していないライブラリが追加されていないか
  - なぜ: 攻撃対象面（attack surface）を不必要に広げる

### 秘密情報
- [ ] **ハードコードされたキー・トークンの検知**
  - APIキー、パスワード、トークンがソースコードに直書きされていないか
  - なぜ: リポジトリ経由で秘密情報が漏洩する。管理方法の詳細は `credential-guard.md` を参照

### ログ出力
- [ ] **機密情報のログ出力禁止**
  - パスワード、トークン、個人情報がログに含まれていないか
  - なぜ: ログ基盤経由で機密情報が拡散し、アクセス管理が困難になる

## 違反検知時の対応

1. PRコメントで指摘（該当箇所をインラインで指定）
2. 重大度を明示: `[CRITICAL]` `[HIGH]` `[MEDIUM]` `[LOW]`
3. 修正方法の具体例を提示
4. 秘密情報のハードコードを検知した場合は `[CRITICAL]` として即時対応を求める
