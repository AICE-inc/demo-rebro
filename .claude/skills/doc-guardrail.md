## 原則

外部に共有するドキュメントには、社内でしか知られるべきでない情報を含めない。
「社外の人がこれを読んだとき、問題になる情報はないか」を基準に判断する。

## チェックリスト

### 個人情報（PII）
- [ ] **氏名・メールアドレス・電話番号の混入チェック**
  - 実在する個人を特定できる情報がドキュメントに含まれていないか
  - なぜ: 個人情報保護法違反のリスク。本人の同意なく公開してはならない
- [ ] **ユーザーデータのサンプル**
  - 例示にはダミーデータを使用し、実データを使わない
  - なぜ: サンプルのつもりで実データを掲載してしまう事故が多い

### 社内情報
- [ ] **内部URL・IPアドレスの露出**
  - 社内Slack、Notion、管理画面のURLが含まれていないか
  - なぜ: 内部システムの存在とアクセス先が外部に知れる
- [ ] **社内用語・コードネームの露出**
  - 未公開のプロジェクト名、社内スラングが含まれていないか
  - なぜ: 未公開情報の間接的な漏洩につながる
- [ ] **社内メンバーのSlack ID・GitHubアカウント**
  - メンションやアカウント名が残っていないか
  - なぜ: ソーシャルエンジニアリングの手がかりになる

### 技術情報
- [ ] **APIキー・エンドポイントの露出**
  - 本番・ステージングのAPIキーやエンドポイントURLが含まれていないか
  - なぜ: 攻撃者にシステムへのアクセス手段を与える
- [ ] **インフラ構成の露出**
  - サーバーのIPアドレス、ポート番号、DB接続情報が含まれていないか
  - なぜ: 攻撃対象の特定を容易にする
- [ ] **セキュリティ対策の詳細**
  - WAFルール、認証フローの内部実装詳細が含まれていないか
  - なぜ: 防御の仕組みを知られると回避策を取られる

## ドキュメント種別ごとの注意点

- **README / 技術ドキュメント**: APIキー・エンドポイントは不要。設定例は仮想データを使用
- **ブログ・外部公開記事**: 社内用語・コードネーム・メンバー名を全削除。技術詳細は公開可能レベルに要約
- **提案書・営業資料**: インフラ構成やセキュリティ対策の詳細を記載しない

## マスキング例

| 種類 | NG | OK |
|------|----|----|
| メール | `satoshi@company.com` | `user@example.com` |
| メール | `yamada@company.com` | `user@example.com` |
| プロジェクト名 | `Project Phoenix（内部コードネーム）` | `新機能` |
| API URL | `https://api.internal.company.com/v2` | `https://api.example.com/v2` |

## 違反検知時の対応

1. 該当箇所を指摘し、上記マスキング例に従って置換を提案
2. 個人情報の場合はダミーデータへの置換例を提示
3. 技術情報の場合は公開可能な抽象度に書き換える例を提示
